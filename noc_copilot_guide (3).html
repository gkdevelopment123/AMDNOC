What You're Actually Building

One sentence: 50 alarms become 1 incident, an AI names the cause, picks a fix, calls the API to apply it, and opens a ticket — live on screen. That arc from chaos to calm is your whole demo.

🔥 The Problem (the pain)

NOC teams get flooded with hundreds of alarms an hour. A single core fault cascades into dozens of downstream alerts. Engineers can't see the signal in the noise → slow root-cause analysis → SLA breaches.

✨ The Solution (the magic)

Specialized AI agents collaborate: they correlate the storm into one incident, reason out the root cause against runbooks, decide a remediation, execute it via tool-calls, and auto-file an ITSM ticket — with a human approval gate on risky actions.

🧠
Why “multi-agent” (not one big prompt)? The brief explicitly says multi-agent, and splitting work into specialists each with a narrow job is what impresses judges and works better. Each agent does one thing well, then hands off.
2

The Agent Team

Five roles. For a half-day build, the three starred (★) carry the whole story — add the rest if time allows.

📡 Ingestion Agent

Takes raw alarms, logs & KPIs and normalizes them into one clean structured schema everything else can read.

🔗 Correlation Agent ★

The highest-value piece. Collapses the alarm storm — 40 alerts → 1 incident — and flags which alarm is the root event vs. cascading noise.

🔍 Root Cause Agent ★

Reasons over the correlated incident + retrieved runbooks (RAG) to name the likely cause with a confidence score.

🔧 Remediation Agent ★

Decides the fix and which tool to call. Auto-executes low-risk fixes; routes high-risk ones to human approval.

🎛️ Orchestrator

The LangGraph state machine that routes an incident through the chain and manages handoffs & branching.

🎫 Action / ITSM Executor

Carries out the chosen tool-calls against the mock router API and mock ITSM, returns real-looking results.

Mocking ITSM & Routers (you have no real access)

This is the right call and judges expect it. A “mock” is a small local service that looks and behaves like the real system so the agent calls it identically.

🎫 Mock ITSM (fake ServiceNow)

A FastAPI service with endpoints like create_ticket, update_ticket, get_ticket. Stores tickets in a dict, returns a ticket ID.

Make it convincing

  • Add realistic latency & status transitions (New → In Progress → Resolved)
  • Show a live “Tickets” panel on the dashboard that updates as tickets are created

🔌 Mock Router / Device API

Fake endpoints like reset_router(device_id), restart_interface(...), clear_bgp(...) returning success/failure JSON.

Make it convincing

  • Occasional simulated failure so the agent must retry or escalate
  • Echo a fake device response so the action looks real on screen
💬
Say this to judges: “ITSM and device APIs are mocked for the hackathon; swapping in real ServiceNow or network controllers is a config change, not a redesign.” That frames the mock as a clean integration boundary — a strength, not a gap.

How to Start — Step by Step

Do these in order inside JupyterLab. Step 2 is non-negotiable first — everything depends on a working endpoint.

Open a Terminal

In JupyterLab: File → New → Terminal. Keep one terminal for the model server, open others for everything else. Most setup happens here, not in notebook cells.

Check if Qwen is already being served

They've hosted the model, so test for it before starting your own:

# does an endpoint already answer?
curl http://localhost:8000/v1/models

Got JSON with a model name? → it's live, note the name, skip to Step 4. Refused/hangs? → start it yourself in Step 3 (and ask organizers the right port).

(If needed) Start the vLLM server

One MI300X = --tensor-parallel-size 1. Run in a dedicated terminal and leave it open:

vllm serve Qwen/Qwen3.6-35B-A3B \
  --host 0.0.0.0  --port 8000 \
  --api-key sk-team-key \
  --tensor-parallel-size 1 \
  --max-model-len 32768 \
  --enable-auto-tool-choice --tool-call-parser qwen3_coder

Replace the model name with whatever's actually on disk (ls ~/.cache/huggingface/hub/). Wait for the “listening on 8000” line. Keep max-model-len modest to respect the 240 GB RAM cap.

Confirm you can talk to it

curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer sk-team-key" \
  -d '{"model":"...","messages":[{"role":"user","content":"say hello"}]}'

A reply means your backend is live. Everything else builds on this.

Install your Python deps

pip install openai langgraph langchain-community \
            chromadb sentence-transformers \
            fastapi uvicorn gradio networkx

Build in this order (always keep a working version)

  • Data: synthetic alarm generator + the scripted storm scenario
  • One agent end-to-end: Root Cause calling the endpoint — prove the loop
  • Mocks: FastAPI ITSM + router APIs
  • Tool-calling: wire the agent to trigger the mocks
  • Correlation: rule-based first, LLM-enhanced if time
  • Dashboard: Gradio tying it together
  • Rehearse the storm demo until it lands every time

Push to GitHub

git init → add a .gitignore that excludes model weights & .cache → commit → push. Your masterplan & prompt files (next section) live here too.

Building It With a Coding Agent

Since you have access to the hosted LLM, you can point a coding agent at it to help build the project. You have two solid options: Aider (uses AMD Qwen directly — fully internal) or Claude Code (stronger reasoning, bridged to AMD via a proxy). Pick one; both use the same three markdown files below.

🧩
Two different "agents" — don't confuse them. The coding agent builds your app (Aider or Claude Code). The NOC agent is your app. You use the first to build the second.
OPTION A

🛠️ Aider → AMD Qwen (fully internal)

Open-source, git-native, points directly at your vLLM endpoint. No external API. 100% on-prem. Runs in a Jupyter terminal.

Best if:

  • You want zero external API calls
  • Hackathon rules require fully internal inference
  • You don't have a Claude API key handy
OPTION B

🤖 Claude Code → AMD vLLM (bridged)

Claude Code's agentic intelligence redirected to your AMD GPU via a litellm proxy. More capable coding agent; your NOC agents still run fully on AMD.

Best if:

  • You want stronger multi-file code reasoning
  • You already use Claude Code locally
  • External API is allowed for the coding layer

🛠️ Option A — Aider + AMD Qwen

One command setup. Points directly at localhost:8000/v1. No proxy needed. Fully internal.

🤖 Option B — Claude Code + AMD vLLM Bridge

Claude Code redirected to your AMD GPU via a lightweight litellm proxy. Your NOC agents still call localhost:8000 directly — only the coding assistant uses the bridge.

⚠️
Why a proxy? vLLM speaks OpenAI format. Claude Code expects Anthropic Messages format. litellm translates between them in one command — no code changes needed on either side.

Confirm the AMD vLLM endpoint is live

# Must return JSON with model name before anything else
curl http://localhost:8000/v1/models
# Note the exact model name — you need it in step 3

Install litellm proxy

pip install 'litellm[proxy]'

Start the litellm bridge (dedicated terminal tab)

Open a new terminal tab in JupyterLab. Keep this running the whole session.

# Replace Qwen3-35B with the model name from step 1
litellm --model openai/Qwen3-35B \
  --api_base http://localhost:8000/v1 \
  --api_key sk-team-key \
  --port 4000

# Verify bridge is up:
curl http://localhost:4000/health

Point Claude Code at the bridge

Open a third terminal tab. Set env vars then launch:

export ANTHROPIC_BASE_URL="http://localhost:4000"
export ANTHROPIC_API_KEY="sk-team-key"
export ANTHROPIC_AUTH_TOKEN="sk-team-key"
export ANTHROPIC_DEFAULT_SONNET_MODEL="Qwen3-35B"
export ANTHROPIC_DEFAULT_HAIKU_MODEL="Qwen3-35B"
export ANTHROPIC_DEFAULT_OPUS_MODEL="Qwen3-35B"
export CLAUDE_CODE_DISABLE_EXPERIMENTAL_BETAS="1"
claude

Claude Code now uses Qwen on AMD MI300X as its brain — fully on-prem.

📂 The three files both agents need

Keep these in the repo root. Works identically for Aider and Claude Code. They let any agent session understand the project and resume where the last one stopped.

MASTERPLAN.md

The single source of truth. What you're building, the architecture, the agent team, the tech stack, the constraints (internal-only, 1 GPU, 240 GB cap), and the build order. The agent reads this first, every session.

Contains

  • Project goal & demo arc
  • Architecture diagram (text)
  • Component list & endpoints
  • Non-negotiable constraints
  • Definition of "done"
PROMPTS.md

The actual system prompts for each NOC agent — Correlation, Root Cause, Remediation — plus the output JSON schemas. This is both your app's brain and a spec the coding agent implements against.

Contains

  • Per-agent system prompt
  • Expected input / output schema
  • Tool definitions (router, ITSM)
  • Few-shot examples
HANDOVER.md

The living progress log. The agent updates this at the end of each session so the next day / next session knows what's done and what's left. This is the file that makes multi-session work possible.

Contains

  • ✓ Done / 🔄 In progress / ⬜ To do
  • Last working state & how to run it
  • Known issues / blockers
  • Next concrete step
📝
The handover rule (works for both agents): end every session by telling the agent "update HANDOVER.md with what we did, the current working state, and the next step." Start every session with "read MASTERPLAN.md and HANDOVER.md before doing anything." That loop is what keeps a multi-day build coherent.

My Suggestions (read before you start)

Things that win hackathons and things that quietly sink them.

✅ Do

  • Build the scripted storm first. One root fault → cascade. Rehearse it so it lands every single time.
  • Add a human-approval gate before destructive actions (router reset). NOC teams care; it shows maturity.
  • Show the correlation visually — 40 alarms collapsing to 1 is your strongest 5 seconds.
  • Keep a working end-to-end version at all times. Simple-but-working beats fancy-but-broken at demo o'clock.
  • Lean on the AMD angle: “local inference on AMD MI300X via ROCm, fully on-prem, no external API” — telecoms love that.

🚫 Avoid

  • Don't load a model in your notebook when vLLM already serves one. OOM risk + wasted time.
  • Don't use share=True on Gradio — it's an external tunnel.
  • Don't attempt real fine-tuning here — it eats your whole time budget & 25 GB storage. Mention it as roadmap.
  • Don't build all 5 agents before one works end-to-end. Depth-first, not breadth-first.
  • Don't let the agent build unattended. Review every change; keep a human steering.
🎯
Scope check: if this is a half-day event, ship Correlation + Root Cause + Remediation + mocks + dashboard. If it's multi-day, add Ingestion, RAG depth, the approval gate, and a polished UI. Either way the storm demo is the centerpiece.
