The Problem & Our Solution

When a core network device fails, a NOC team is buried under dozens of cascading alarms. Diagnosing and fixing it manually is slow and SLA-breaching. Our Copilot does it autonomously in seconds.

😭 Without the Copilot

  • 33+ alarms flood in across 17 devices — which is the real cause?
  • Engineers manually correlate events under pressure
  • Root-cause analysis takes 20+ minutes
  • Remediation steps run by hand, ticket raised late
  • SLA breaches; customer-facing outages drag on

🚀 With the Copilot

  • 33 alarms collapse into 1 root incident automatically
  • Root cause identified with confidence + evidence
  • Safe fixes auto-execute; risky ones wait for one click
  • ITSM ticket opened and managed end-to-end
  • Right teams alerted; full lifecycle in under a minute

End-to-End Flow

What happens, in order, when an outage is simulated — each stage is a real call to the local model, streamed live to the dashboard.

1
Ingestion

Alarm storm arrives

~33 alarms across 17 devices (core router, PE routers, switches, base stations) stream in with severities, KPIs and timestamps.

2
Correlation Agent

Collapse into one incident

The agent finds the common root and groups all symptomatic alarms into a single incident — the correlation map visibly converges to one hub.

3
RAG + Reranker

Retrieve & rank knowledge

Five candidate runbooks are retrieved by vector similarity, then the LLM reranks them; the Knowledge Base panel shows scores and the best match.

4
Root Cause Agent

Diagnose the fault

Produces the root cause with a confidence bar and evidence bullets drawn from the alarms and the top-ranked runbook.

5
Remediation Agent + Execution

Plan & act

A risk-ranked plan is produced. Low-risk steps (clear BGP, restart interface) auto-execute against the mock device API; the high-risk router reset is held with an 🔒 APPROVAL gate.

6
ITSM Agent

Open the ticket

A ServiceNow-style incident ticket is created (priority by severity) and appears live on the ITSM board with full details and a work-notes history.

7
Alerting Agent

Notify the right people

Notifications are routed by severity/impact to on-call, NOC Tier-2 and management, with an escalation badge when customer-facing sites are hit.

8
Human-in-the-loop

Approve & resolve

The engineer clicks Approve & Execute to run the risky step (ticket → In Progress), then confirms resolution by chat (ticket → Resolved with an auto-generated note).

💡 The full lifecycle a ticket moves through: New (auto-created) → In Progress (human approves the high-risk action) → Resolved (confirmed by chat), every transition logged in the audit trail and reflected live on the ITSM board.

Ingestion & the Production Path

How alarms, logs and KPIs enter the system today — and exactly how this connects to a real network in production. The ingestion layer is decoupled: only the source changes, the entire multi-agent pipeline stays identical.

📥 The alarm schema (production-shaped)

Every alarm — whether from the simulator today or a live feed tomorrow — is one object with the same fields a real NOC fault-management system produces:

{
  "alarm_id":    "ALM-001",
  "timestamp":   "2026-06-16T09:00:00",
  "device_id":   "Router-Core-01",
  "device_type": "core",
  "site":        "DC-North",
  "alarm_type":  "BGP_PEER_DOWN",
  "severity":    "CRITICAL",
  "description": "BGP session lost with upstream peer 10.0.0.1...",
  "kpis":        { "cpu": 94, "packet_loss": 0, "latency_ms": 0 }
}

Paired with a topology model (device upstream/downstream relationships) so the Correlation Agent can understand cascades. Both are handed to the pipeline as one JSON payload.

🎲 Today · Simulated source

A realistic alarm generator produces a scripted outage — a root fault on the core router cascading to PE routers, switches and base stations over ~60 seconds. The inputs are synthetic, but the schema and the AI processing are real: every alarm flows through the same pipeline a live feed would.

alarms = generate_storm()      # the swap point
payload = {"alarms": alarms, "topology": topo}
correlate(payload)             # → AI pipeline

🏭 Production · Live source

Replace the single source line with a thin ingestion adapter that subscribes to the real feed and maps events into the same alarm object. Nothing downstream changes.

alarms = ingest_from_kafka()   # or SNMP / EMS
payload = {"alarms": alarms, "topology": cmdb}
correlate(payload)             # same pipeline

🔌 Real-world ingestion connectors

📡 SNMP Traps

Devices emit traps to a collector; the adapter parses them into alarm objects.

📝 Syslog

Stream device logs to a syslog server; critical/error lines become alarms.

📊 Streaming Telemetry

gNMI/streaming KPIs (CPU, packet loss, latency) trigger threshold-breach alarms.

📫 Kafka / Event Bus

EMS/devices publish events to a topic; the pipeline consumes it in real time at scale.

🌐 OSS / EMS API

Adapter to existing fault management (Netcool, U2000, NSP) — the usual telecom case.

🗃️ CMDB / Inventory

Live topology pulled from the CMDB instead of a static file.

🚀 What changes for production (and what doesn't)

Swap these adapters

  • ⇄ Alarm source → Kafka / SNMP / EMS feed
  • ⇄ Mock ITSM API → real ServiceNow REST connector
  • ⇄ Mock router API → Ansible / NETCONF / vendor automation
  • ⇄ Static topology → live CMDB sync
  • ⇄ Click-to-run → continuous streaming mode

Stays exactly the same

  • ✓ All 5 agents & their prompts
  • ✓ Correlation → RCA → remediation logic
  • ✓ RAG knowledge base & reranking
  • ✓ Human-in-the-loop approval gate
  • ✓ The on-prem local model on MI300X
💡 The one-line pitch: "Ingestion is decoupled — today a realistic simulator, in production a thin adapter to your existing EMS, Kafka or SNMP feed. The same alarm schema flows into the same multi-agent pipeline, so going live means writing one connector, not rebuilding the system."

How to Run It

Two terminals: one for the model, one for the app.

0

One-time: pin the RAG dependency

pip install "huggingface-hub>=0.34.0,<1.0"
1

Terminal 1 — start the local LLM (vLLM)

# serves Qwen3-Coder-30B on port 8000
vllm serve Qwen/Qwen3-Coder-30B-A3B-Instruct \
  --host 0.0.0.0 --port 8000 \
  --tensor-parallel-size 1 \
  --max-model-len 100000 \
  --enable-auto-tool-choice \
  --tool-call-parser qwen3_xml \
  --disable-log-stats

Verify: curl -s http://localhost:8000/v1/models

2

Terminal 2 — start all app services

# starts mocks + ITSM board + admin + dashboard, health-checks each
bash run_all.sh

Leaves vLLM untouched. Prints the dashboard link plus the ITSM board (8080) and admin (8090) URLs.

3

Open & demo

Open the dashboard link, click ⚡ Simulate P1 Outage (or 🎲 Simulate Random Incident), watch the pipeline run, click 🔒 Approve & Execute, open the ITSM board, and resolve via the Copilot chat.

🔄 Stop / restart: press Ctrl+C in the run_all.sh terminal and run bash run_all.sh again. Keep vLLM running. If the in-app links point to the wrong host, edit PROXY_BASE at the top of app.py.
